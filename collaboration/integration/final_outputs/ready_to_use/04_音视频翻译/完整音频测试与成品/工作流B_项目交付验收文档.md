## 模块2：业务链路逻辑说明

### 2.1 双模式自动分流机制

工作流入口为 `auto_mode_judge` 节点（智能模式判断），根据输入文件类型自动分流：

| 输入条件 | 触发模式 | 分流逻辑 |
|---------|---------|---------|
| 仅上传音频文件（media_file） | 模式一 | 自动执行预处理流程 |
| 上传 Excel 文件（finalized_excel） | 模式二 | 自动执行回填流程 |

### 2.2 模式二原始音频复用机制

**核心问题**：模式二需要原始音频进行音视频混音，但用户不应重复上传。

**存储方案**：
- 模式一运行时，`asr_recognition` 节点将原始音频上传至对象存储，URL 写入全局变量 `original_audio_url`
- 同时通过 Supabase 会话存储关联 `user_id` → `original_audio_url`
- 模式二运行时，`auto_mode_judge` 节点优先从全局变量读取 `original_audio_url`；若为空，则从 Supabase 按 `user_id` 查询最近一次上传的音频 URL

**关联复用流程**：
```
模式一：上传音频 → ASR识别 → 音频URL存入GlobalState.original_audio_url + Supabase
                                    ↓
模式二：上传Excel → auto_mode_judge自动查询original_audio_url → 透传给batch_tts/media_compile
```

**边界限制**：
- 全新会话（无模式一历史）没有音频缓存，模式二无法独立运行
- 同一会话内，模式一上传的音频可被模式二自动复用，无需重复上传

### 2.3 模式二末尾并行双通道

模式二全部 TTS 音频合成完毕后，`end_mode2` 节点触发并行双通道：

```
end_mode2
  ├── 通道A：qr_code_generation（二维码生成）
  └── 通道B：excel_output_generate（四列Excel输出）
       ↓ 两个通道并行执行，互不阻塞
      END
```

**通道A — 二维码生成**：
- 输入：`final_media_url`（合并音频）、`segment_audio_urls`（分段音频列表）
- 输出：二维码 PNG 图片，编码内容为播放页面 URL（含 `audioList` 参数，加载全部音频）
- 关键约束：二维码必须编码全部音频 URL，不可仅返回第一条

**通道B — 四列 Excel 输出**：
- 输入：`table_data`（原始 Excel 三列数据）、`segment_audio_urls`（分段音频 URL 列表）
- 输出：四列对照 Excel（音频文字 / 机器译文 / 人工审核 / 音频下载地址）
- 匹配规则：按行一一对应，每行匹配对应分句独立 TTS 音频链接

### 2.4 完整执行链路

**模式一**：
```
auto_mode_judge → asr_recognition → data_table_construct → batch_translation
→ excel_generate → excel_data_fill → end_mode1 → END
```

**模式二**：
```
auto_mode_judge → excel_read → data_validate → batch_tts → media_compile
→ end_mode2 → [并行] qr_code_generation + excel_output_generate → END
```

## 模块5：正式验收判定标准

### 5.1 译文中文残留约束

| 判定项 | 合格标准 | 不合格判定 |
|-------|---------|-----------|
| 机器译文列 | 纯英文，无中文字符（标点除外） | 出现中文字符即为不合格 |
| 人工审核列 | 纯英文，无中文字符 | 出现中文字符即为不合格 |
| LLM 推理噪声 | 机器译文列不得包含 LLM 推理过程（如"哦不对""重新数一下"等） | 出现自问自答、数字编号列表等推理噪声即为不合格 |

### 5.2 Excel 分句行数与音频严格对齐

| 判定项 | 合格标准 |
|-------|---------|
| 行数对齐 | Excel 有效行数（人工审核或机器译文非空）= TTS 合成音频段数 = `segment_audio_urls` 长度 |
| 不漏句 | 每一行有效文本必须生成对应 TTS 音频，不得跳过 |
| 不多生成 | TTS 音频段数不得多于 Excel 有效行数 |
| 顺序一致 | `segment_audio_urls` 按 `orig_idx` 排序，与 Excel 行顺序一一对应 |

### 5.3 文件格式保留规范

| 文件类型 | 格式要求 |
|---------|---------|
| Excel | `.xlsx` 格式，UTF-8 编码，单元格开启多行文本 |
| 音频 | `.mp3` 格式，采样率 24000Hz |
| 二维码 | `.png` 格式，RGB 色彩模式 |

### 5.4 统一文件命名规范

| 文件类型 | 命名规则 | 示例 |
|---------|---------|------|
| 模式一输出 Excel | `Fan_Yi_Dai_Shen_Xiao_{时间戳}_{随机后缀}.xlsx` | `Fan_Yi_Dai_Shen_Xiao_20260714_223109_abc123.xlsx` |
| 模式二输出 Excel | `Fan_Yi_Cheng_Pin_Shu_Chu_{时间戳}_{随机后缀}.xlsx` | `Fan_Yi_Cheng_Pin_Shu_Chu_20260714_223109_def456.xlsx` |
| 分段音频 | `segment_{序号}_{随机后缀}.mp3` | `segment_000_abc123.mp3` |
| 合并音频 | `merged_{句数}sentences_normalized_{随机后缀}.mp3` | `merged_4sentences_normalized_abc123.mp3` |
| 二维码 | `qr_code_image_{随机后缀}.png` | `qr_code_image_abc123.png` |

### 5.5 程序异常/部分失败时输出规则

| 异常场景 | 处理策略 | 输出规则 |
|---------|---------|---------|
| TTS 部分段失败 | 软失败 + 静默补位 | 已成功段正常输出，失败段用对应长度静音补位，`message` 提示"X 段 TTS 失败，已用静音补位" |
| TTS 月度配额耗尽（code=4036） | 部分降级 | 返回已成功部分的拼接音频 + 失败 segment_id 列表 + 友好提示 |
| TTS 全部失败 | 抛友好异常 | 保留原始错误信息 + 解决方式说明 |
| Excel 预签名 URL 过期（HTTP 403） | 立即终止重试 | 提示"请重新上传 Excel 文件" |
| 翻译 LLM 输出推理噪声 | 自动清洗 | `_clean_translation_output()` 自动提取纯英文译文 |

### 5.6 TTS 音频验收标准

| 判定项 | 合格标准 |
|-------|---------|
| 音色统一 | 全部段落使用同一音色（Allison 美式女声），无音色割裂 |
| 口音统一 | 全程美式发音，无中英混合口音 |
| 语速稳定 | 统一 17.0 字/秒（speed_ratio 1.0），各段语速一致 |
| 段间停顿 | 每段之间 0.4 秒静音停顿（纪录片风格） |
| 无 SSML 标签朗读 | 音频中不得出现"speak""lang"等 SSML 标签被朗读的情况 |

### 5.7 二维码功能验收

| 判定项 | 合格标准 |
|-------|---------|
| 全部音频加载 | 二维码扫码后必须加载全部音频（`audioList` 参数包含所有 `segment_audio_urls`），不可仅返回第一条 |
| 可正常播放 | 扫码后播放页可正常播放每段音频 |
| 图片质量 | PNG 格式，box_size=10，60px 白边，可正常扫码识别 |
| 有效期验证 | 二维码内嵌音频链接 24 小时内可正常扫码播放；超过 24 小时应返回 403 |
| 重新运行验证 | 重新运行工作流后，旧二维码的音频链接立即失效，新二维码可正常播放 |

### 5.8 并行通道验收

| 判定项 | 合格标准 |
|-------|---------|
| 双通道并行 | 二维码生成（通道A）与四列 Excel 输出（通道B）并行执行，互不阻塞 |
| 双产物输出 | 模式二结束后，`qr_code_url` 和 `excel_output_url` 均非空 |
| 四列 Excel 内容 | 前三列取自输入 Excel 原数据，第四列为对应分句独立 TTS 音频链接（非合并音频链接） |
| 行对应 | Excel 行数 = 分段音频数，一一对应 |

**安全规范**：禁止将 Token 暴露在 Uniapp 前端代码，统一由后端服务中转请求。

### 2、接口选型规范

**1）同步接口【执行工作流】**

- 底层强制限制：最大运行时长 120 秒，超时强制断开连接。
- 适用场景：短音频调试；**不支持 17MB 长音频处理，正式业务禁止使用**。

**2）异步方案【正式环境唯一推荐】（规避 120 秒超时限制）**

链路顺序：

1. 调用【异步执行工作流】提交任务 → 立刻返回 `task_id`
2. 后端定时轮询调用【查询异步执行任务】
   - 状态：处理中 → 继续等待轮询
   - 状态：成功 → 获取全部输出数据（音频链接、二维码等）
   - 状态：失败 → 获取错误信息

### 3、入参两种业务模式

**模式一（首次上传原始音频，流程一）**

传入参数：`media_file` 原始音视频文件对象

**模式二（人工校对完成 Excel 二次生成音频，流程二）**

传入参数：`finalized_excel` + `original_audio_url`

特性：自动复用初次上传的音频地址，无需重复上传音频文件，实现变量回填。

### 4、废弃接口标注

**流式执行工作流**：当前项目业务不需要，开发无需对接。

## API Token 分发与安全规范

1. API Token 需要交付项目后端开发人员单独保存。
2. **安全红线**：禁止将 Token 暴露在 Uniapp 前端代码、移动端客户端内。
3. **标准调用架构**：Uniapp 前端发起请求 → 自有后端服务中转 → 后端携带 Token 访问扣子工作流 API。
4. **风险提示**：Token 一旦外泄，外部人员可随意调用本工作流，消耗算力资源；怀疑泄露时，可删除旧 Token 并重新创建。
